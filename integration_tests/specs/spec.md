# ChronoDB 集成测试规范

## 概述

本文档定义了 ChronoDB 集成测试的规范，包括测试目标、测试范围、测试用例设计和验收标准。

## 测试目标

1. **功能验证**: 验证 ChronoDB 的写入、查询、聚合等功能是否正常工作
2. **兼容性验证**: 验证 ChronoDB 与 Prometheus API 的兼容性
3. **性能评估**: 评估 ChronoDB 在不同数据规模下的性能表现
4. **可靠性验证**: 验证 ChronoDB 在故障场景下的恢复能力

## 测试范围

### 单机模式测试
- 基本写入功能
- 基本查询功能
- 聚合查询
- 范围查询
- 元数据查询

### 分布式模式测试
- 集群部署
- 数据分片
- 故障转移
- 数据一致性

### 对比测试
- 与 Prometheus 的 API 兼容性
- 查询结果一致性
- 性能对比

## 数据规模定义

| 规模 | 指标数 | 序列数/指标 | 样本数/序列 | 时间范围 |
|------|--------|-------------|-------------|----------|
| Small | 10 | 10 | 100 | 1小时 |
| Medium | 50 | 50 | 1,000 | 6小时 |
| Large | 100 | 100 | 10,000 | 24小时 |

## 测试用例

### 1. 写入测试

#### TC-001: 单条数据写入
- **描述**: 验证单条指标数据写入
- **步骤**:
  1. 构造单条指标数据
  2. 调用写入 API
  3. 查询验证数据已写入
- **预期结果**: 数据写入成功，查询返回正确值

#### TC-002: 批量数据写入
- **描述**: 验证批量指标数据写入
- **步骤**:
  1. 构造批量指标数据（1000条）
  2. 调用写入 API
  3. 查询验证数据已写入
- **预期结果**: 所有数据写入成功

#### TC-003: 不同类型指标写入
- **描述**: 验证 Gauge、Counter、Histogram 类型指标写入
- **步骤**:
  1. 构造不同类型指标数据
  2. 调用写入 API
  3. 查询验证数据已写入
- **预期结果**: 所有类型数据写入成功

### 2. 查询测试

#### TC-004: 即时查询
- **描述**: 验证即时查询功能
- **步骤**:
  1. 写入测试数据
  2. 执行即时查询
  3. 验证返回结果
- **预期结果**: 返回正确的即时数据

#### TC-005: 范围查询
- **描述**: 验证范围查询功能
- **步骤**:
  1. 写入测试数据
  2. 执行范围查询
  3. 验证返回结果
- **预期结果**: 返回正确的时间序列数据

#### TC-006: 聚合查询
- **描述**: 验证聚合查询功能
- **步骤**:
  1. 写入测试数据
  2. 执行聚合查询（sum、avg、max、min、count）
  3. 验证返回结果
- **预期结果**: 返回正确的聚合结果

### 3. PromQL 算子测试

#### TC-007: 范围向量函数
- **描述**: 验证 rate、irate、increase 等函数
- **测试函数**: rate, irate, increase, delta, idelta
- **预期结果**: 函数计算结果正确

#### TC-008: 数学函数
- **描述**: 验证 abs、ceil、floor 等函数
- **测试函数**: abs, ceil, floor, round, sqrt
- **预期结果**: 函数计算结果正确

#### TC-009: 二元运算符
- **描述**: 验证 +、-、*、/ 等运算符
- **预期结果**: 运算结果正确

### 4. 对比测试

#### TC-010: 查询结果对比
- **描述**: 对比 Prometheus 和 ChronoDB 的查询结果
- **步骤**:
  1. 向两者写入相同数据
  2. 执行相同查询
  3. 对比返回结果
- **预期结果**: 结果差异在容差范围内（值容差 1%，时间容差 1s）

### 5. 故障注入测试

#### TC-011: 容器重启测试
- **描述**: 验证容器重启后数据完整性
- **步骤**:
  1. 写入测试数据
  2. 重启容器
  3. 查询验证数据完整
- **预期结果**: 数据不丢失，查询正常

#### TC-012: 随机故障测试
- **描述**: 验证系统在随机故障下的稳定性
- **步骤**:
  1. 启动故障注入
  2. 持续执行查询
  3. 验证系统可用性
- **预期结果**: 系统能够自动恢复，查询成功率 > 80%

## 验收标准

### 功能验收标准
- 所有写入测试用例通过率 100%
- 所有查询测试用例通过率 100%
- 所有 PromQL 算子测试用例通过率 ≥ 95%

### 性能验收标准
- 写入吞吐量 ≥ 10,000 样本/秒
- 查询延迟 P99 < 1秒（小规模数据）
- 查询延迟 P99 < 5秒（中规模数据）

### 可靠性验收标准
- 容器重启后数据不丢失
- 故障注入期间查询成功率 ≥ 80%
- 系统能够自动从故障中恢复

### 兼容性验收标准
- 与 Prometheus API 的查询结果匹配率 ≥ 99%
- 支持的 PromQL 算子覆盖率 ≥ 95%

## 测试环境要求

### 硬件要求
- CPU: 4核及以上
- 内存: 8GB及以上
- 磁盘: 50GB可用空间

### 软件要求
- Docker 20.10+
- Python 3.8+
- 本地镜像: prom/prometheus:latest

### 网络要求
- 本地网络访问
- 端口 9090-9092 可用

## 测试报告

测试完成后应生成以下报告：
1. **JSON 报告**: 包含详细的测试结果数据
2. **HTML 报告**: 可视化的测试报告
3. **Markdown 报告**: 便于阅读和分享

报告内容应包括：
- 测试汇总统计
- 各测试套件结果
- 失败的测试用例详情
- 性能对比数据
- 故障注入测试结果
