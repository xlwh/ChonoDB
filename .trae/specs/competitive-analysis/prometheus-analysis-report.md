# Prometheus 深入调研分析报告

## 1. 系统概述

### 1.1 背景

Prometheus 是由 SoundCloud 开发的开源监控和告警系统，于 2016 年加入 CNCF（云原生计算基金会），成为继 Kubernetes 之后的第二个毕业项目。Prometheus 专注于监控和告警，采用拉取模式采集数据，具有强大的 PromQL 查询语言，并与 Grafana 深度集成。

### 1.2 核心特点

- **多维数据模型**：时间序列由指标名称和键值对（标签）标识
- **灵活的查询语言**：PromQL 支持强大的数据查询和转换能力
- **拉取模式**：主动从配置的目标拉取指标数据
- **独立部署**：每个 Prometheus 服务器独立运行，仅依赖本地存储
- **服务发现**：支持多种服务发现机制（Kubernetes、Consul、DNS 等）
- **告警管理**：通过 Alertmanager 实现告警去重、分组、路由和静默

### 1.3 适用场景

- **云原生监控**：Kubernetes 环境下的容器监控
- **应用性能监控**：微服务架构的性能监控
- **基础设施监控**：服务器、网络、存储等基础设施监控
- **告警和通知**：实时告警和事件通知

## 2. 架构分析

### 2.1 整体架构

Prometheus 采用分层架构设计，主要包括以下组件：

```

│  │ (数据采集)   │  │ (时序存储)   │  │ (查询接口)   │     │
│  └──────────────┘  └──────────────┘  └──────────────┘     │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐     │
│  │ PromQL       │  │ Rules        │  │ Service      │     │
│  │ Engine       │  │ Engine       │  │ Discovery    │     │

└─────────────────────────────────────────────────────────────┘
```

### 2.2 核心组件

#### 2.2.1 TSDB（时序数据库）

**存储架构**：

1. **Head Block**：
   - 内存中的实时数据块
   - 存储最近的数据（默认 2 小时）
   - 使用 WAL（Write-Ahead Log）保证数据持久性
   - 数据结构：`stripeSeries`（活跃系列）、`memSeries`（内存系列）

2. **WAL（Write-Ahead Log）**：
   - 存储在 `wal` 目录，128MB 分段
   - 包含未压缩的原始数据
   - 保留至少 3 个 WAL 文件
   - 高流量服务器可能保留更多 WAL 文件以保持至少 2 小时的原始数据

3. **Persistent Blocks**：
   - 持久化数据块，按时间分片
   - 每个 Block 包含：
     - `chunks/`：数据块文件
     - `index`：索引文件
     - `meta.json`：元数据
   - Block 大小：默认 2 小时

4. **Compaction（压缩合并）**：
   - 自动运行的后台进程
   - 将多个小 Block 合并为大 Block
   - 清理已删除的数据
   - 优化存储空间和查询性能

**数据压缩**：

- **Gorilla 压缩算法**：
  - Facebook 开发的时间序列压缩算法
  - Prometheus 2.0+ 完全使用 Gorilla（块结构略有调整）
  - 压缩比：约 1.37 字节/样本
  - 支持三种编码类型：
    - Type 0：兼容旧版本
    - Type 1：更好的压缩行为（推荐）
    - Type 2：可变位宽编码

**存储性能**：

- 写入吞吐量：约 1-2 百万样本/秒
- 压缩比：10-15x
- 存储效率：Gorilla 压缩 + 块压缩

#### 2.2.2 PromQL 查询引擎

**查询优化策略**：

1. **查询性能优化**：
   - 限制时间序列选择：使用选择性标签匹配器减少基数
   - 使用适当的时间范围：避免过长时间范围查询
   - 避免高基数标签：消除不必要的标签
   - 减少嵌套聚合层级：避免多层 `sum by()` 嵌套

2. **查询性能改进**（2025 年最新）：
   - PromQL join 性能提升 >50%
   - 内存分配减少高达 96%
   - 标签集长度优化：针对数百字节长的标签集

3. **Recording Rules**：
   - 预计算复杂查询，减少运行时负载
   - 命名规范：`level:metric:operations`
   - 定期评估并存储结果
   - 最佳实践：为非紧急告警添加 `for` 子句（1-10 分钟）

**查询性能瓶颈**：

- 单线程 PromQL 执行
- 无查询优化器
- 全表扫描式索引
- 查询长时间段数据时需要处理海量原始样本

#### 2.2.3 服务发现机制

**支持的服务发现类型**：

1. **静态配置**：手动指定监控目标地址和端口
2. **动态服务发现**：
   - Kubernetes：自动发现 Pod、Service、Node
   - Consul：服务注册中心
   - DNS：DNS SRV 记录
   - EC2、Azure、GCE：云平台实例
   - 文件：基于配置文件的服务发现

**服务发现流程**：

1. Prometheus 通过服务发现机制获取目标列表
2. 定期更新目标列表（动态环境）
3. 对每个目标执行健康检查
4. 通过 `/metrics` 端点拉取指标数据

#### 2.2.4 Alertmanager

**核心功能**：

1. **去重（Deduplication）**：
   - 避免重复告警
   - 基于告警标签进行去重

2. **分组（Grouping）**：
   - 将相似告警分组
   - 减少告警噪音
   - 按标签分组（如 `alertname`、`cluster`）

3. **路由（Routing）**：
   - 根据标签将告警路由到不同接收器
   - 支持邮件、Slack、PagerDuty、钉钉等

4. **静默（Silencing）**：
   - 临时屏蔽特定告警
   - 支持时间范围和标签匹配

5. **抑制（Inhibition）**：
   - 当某些告警触发时，抑制其他告警
   - 避免告警风暴

**告警流程**：

1. Prometheus 检测阈值违规，触发告警事件
2. 事件推送至 Alertmanager
3. Alertmanager 进行去重、分组、路由
4. 通过配置的通知渠道发送告警

### 2.3 联邦集群架构

**联邦模式**：

Prometheus 联邦是实现"层级化监控数据聚合"的核心机制，通过"上游 Prometheus 采集细分数据，下游 Prometheus 聚合全局数据"，解决大规模或跨层级监控的"数据分散"问题。

**联邦类型**：

1. **层级联邦（Hierarchical Federation）**：
   - 本地 Prometheus 服务器从服务采集指标
   - 应用 Recording Rules 降低基数
   - 通过 `/federate` 端点暴露聚合系列
   - 联邦服务器向上抓取选定的聚合指标
   - 写入长期存储和仪表板

2. **跨服务联邦（Cross-service Federation）**：
   - 一个服务的 Prometheus 服务器配置为从另一个服务的 Prometheus 服务器抓取选定数据
   - 支持在单个服务器内对两个数据集进行告警和查询

**联邦配置**：

```yaml
scrape_configs:
  - job_name: 'federate'
    scrape_interval: 15s
    honor_labels: true
    metrics_path: '/federate'
    params:
      'match[]':
        - '{job="prometheus"}'
        - '{__name__=~"job:.*"}'
    static_configs:
      - targets:
        - 'source-prometheus-1:9090'
        - 'source-prometheus-2:9090'
```

**联邦局限性**：

- 仍然是基于拉取的抓取
- 不添加集群、共享存储或自动分片
- 是一种受控的层次结构组合方式
- 复杂性高，需要精心设计

### 2.4 远程存储集成

**Remote Write**：

- Prometheus 将采集的样本写入远程存储
- 支持多种远程存储后端（Thanos、Cortex、VictoriaMetrics 等）
- 配置示例：

```yaml
remote_write:
  - url: "http://remote-storage:9090/api/v1/write"
    queue_config:
      max_samples_per_send: 1000
      max_shards: 200
```

**Remote Read**：

- 从远程存储读取数据
- 支持全局视图和联邦设置
- 提供可扩展性和数据持久性

**优势**：

- 全局视图/联邦：从其他 Prometheus 实例或兼容的远程存储查询数据
- 可扩展性：通过卸载存储，Prometheus 实例可以处理更高的摄取速率或更长的保留期
- 数据持久性和冗余：在外部系统中存储数据可以提供比单独本地磁盘存储更好的数据持久性和冗余选项

## 3. 功能分析

### 3.1 数据写入

**写入协议**：

- **Pull 模式**：Prometheus 主动从目标拉取指标
- **文本格式**：标准 Prometheus 文本格式
- **Protocol Buffers**：更高效的二进制格式

**写入性能**：

- 写入吞吐量：约 1-2 百万样本/秒
- 写入延迟：低（本地存储）
- 写入稳定性：高（WAL 保证）

**写入优化**：

- 批量写入：通过 WAL 批量写入
- 异步写入：后台异步刷盘
- 压缩写入：Gorilla 压缩减少存储空间

### 3.2 数据查询

**PromQL 功能**：

1. **即时查询**：查询单个时间点的数据
2. **范围查询**：查询时间范围内的数据
3. **聚合函数**：sum、avg、min、max、count、stddev、stdvar、topk、bottomk
4. **时间函数**：rate、irate、delta、increase、predict_linear
5. **数学函数**：abs、ceil、floor、round、sqrt、log、exp
6. **布尔运算**：==、!=、>、<、>=、<=
7. **集合运算**：and、or、unless

**查询性能**：

- 查询延迟：取决于查询复杂度和数据量
- 查询吞吐量：受单线程限制
- 查询优化：Recording Rules、查询缓存

### 3.3 数据管理

**数据保留**：

- 默认保留时间：15 天
- 可配置保留时间：`--storage.tsdb.retention.time`
- 可配置存储空间：`--storage.tsdb.retention.size`

**数据备份**：

- 手动备份：复制数据目录
- 快照备份：`POST /api/v1/admin/tsdb/snapshot`
- 远程备份：通过 Remote Write 同步到远程存储

**数据恢复**：

- 从快照恢复
- 从备份目录恢复
- WAL 重放恢复

### 3.4 监控告警

**告警规则**：

```yaml
groups:
  - name: example
    rules:
      - alert: HighRequestLatency
        expr: job:request_latency_seconds:mean5m{job="myjob"} > 0.5
        for: 10m
        labels:
          severity: page
        annotations:
          summary: High request latency
          description: description info
```

**告警最佳实践**：

1. **防抖设计**：非紧急告警添加 `for` 子句（1-10 分钟）
2. **标签规范**：统一 `severity` 级别（critical > warning > info）
3. **分组策略**：按 `alertname`、`cluster`、`service` 分组
4. **路由策略**：根据严重程度路由到不同接收器

### 3.5 可视化

**Grafana 集成**：

- Prometheus 作为 Grafana 数据源
- 丰富的图表类型：折线图、柱状图、饼图、仪表盘
- 告警面板：显示当前告警状态
- 变量支持：动态查询参数

**内置 UI**：

- 查询界面：PromQL 查询编辑器
- 图形展示：简单的图形展示
- 目标状态：显示抓取目标状态
- 配置查看：查看当前配置

## 4. 性能分析

### 4.1 性能指标

| 性能指标 | Prometheus | 备注 |
|---------|-----------|------|
| 写入吞吐量 | 1-2 百万样本/秒 | 单节点性能 |
| 查询延迟 | 取决于查询复杂度 | 单线程执行 |
| 压缩比 | 10-15x | Gorilla 压缩 |
| 存储效率 | 1.37 字节/样本 | Gorilla 压缩 |
| CPU 使用 | 2.8-3.21 核心/百万指标/秒 | Prometheus v2-v3 |

### 4.2 性能瓶颈

1. **单节点限制**：
   - 本地存储限制垂直扩展
   - 超过 10 万活跃时间序列或保留数月数据面临存储和查询性能问题
   - 解决方案：联邦或第三方解决方案（Thanos、Cortex）

2. **WAL 重放性能**：
   - WAL 重放时间约 25 分钟
   - 启动 Prometheus 需要约 30 分钟
   - 单线程 WAL 重放/观察器限制吞吐量

3. **查询性能**：
   - 单线程 PromQL 执行
   - 无查询优化器
   - 全表扫描式索引
   - 查询长时间段数据时需要处理海量原始样本

4. **扩展性限制**：
   - Prometheus 的设计将可扩展性负担放在用户身上
   - 大规模设置不常见，除非有专门的观测团队和大量资源
   - 需要大量工程专业知识、调优和额外工具（如 Thanos 或 Cortex）

### 4.3 性能优化策略

1. **查询优化**：
   - 使用选择性标签匹配器限制基数
   - 使用适当的时间范围
   - 避免高基数标签
   - 减少嵌套聚合层级

2. **Recording Rules**：
   - 预计算复杂查询
   - 减少运行时负载
   - 加速仪表板加载

3. **联邦架构**：
   - 分层聚合降低基数
   - 减少中心服务器负载
   - 提高查询性能

4. **远程存储**：
   - 卸载存储负担
   - 延长数据保留期
   - 提高可扩展性

### 4.4 大规模案例

**Flipkart 案例**：

- 规模：8000 万指标
- 解决方案：层级联邦
- 策略：
  - 本地 Prometheus 服务器从服务采集指标
  - 应用 Recording Rules 降低基数
  - 通过 `/federate` 端点暴露聚合系列
  - 联邦服务器向上抓取选定的聚合指标
  - 写入长期存储和仪表板

**效果**：

- 大幅降低指标基数
- 减少中心服务器负载
- 提高查询性能

## 6. 劣势分析

### 6.1 技术局限

1. **单节点限制**：
   - 本地存储限制垂直扩展
   - 无法水平扩展
   - 存储容量有限

2. **查询性能**：
   - 单线程 PromQL 执行
   - 无查询优化器
   - 全表扫描式索引

3. **存储效率**：
   - Gorilla 压缩比有限（约 1.37 字节/样本）
   - 无列式存储
   - 无数据分层

4. **降采样能力**：
   - 不支持内置降采样
   - 需要额外配置或使用外部组件
   - 查询长时间段数据性能差

5. **分布式能力**：
   - 无内置分布式支持
   - 联邦架构复杂
   - 需要第三方解决方案（Thanos、Cortex）

### 6.2 已知问题

1. **WAL 重放性能**：
   - WAL 重放时间长达 25 分钟
   - 启动时间长（约 30 分钟）
   - 单线程限制吞吐量

2. **高基数问题**：
   - 高基数标签导致性能下降
   - 存储和查询性能受影响
   - 需要精心设计标签策略

3. **长期存储**：
   - 本地存储不适合长期数据保留
   - 需要远程存储解决方案
   - 增加系统复杂性

### 6.3 改进空间

1. **查询优化器**：
   - 引入查询优化器
   - 支持查询并行化
   - 提高查询性能

2. **存储优化**：
   - 支持列式存储
   - 提高压缩比
   - 支持数据分层

3. **分布式架构**：
   - 原生支持分布式
   - 自动数据分片
   - 简化联邦架构

4. **降采样机制**：
   - 内置降采样支持
   - 自动选择合适精度
   - 提高长时间段查询性能

4. **应用性能监控**：
   - 微服务性能监控
   - API 延迟监控
   - 错误率监控

### 7.2 不推荐场景

1. **大规模监控**：
   - 活跃时间序列 > 100 万
   - 需要水平扩展
   - 长期数据保留（> 1 年）

2. **高基数场景**：
   - 标签值数量巨大（如用户 ID）
   - 查询性能要求极高
   - 需要复杂查询优化

3. **分布式存储**：
   - 需要跨数据中心监控
   - 需要全局查询视图
   - 需要高可用存储

### 7.3 替代方案

1. **VictoriaMetrics**：
   - 更高的压缩比
   - 更好的查询性能
   - 兼容 Prometheus 协议

2. **Thanos**：
   - 无限数据保留
   - 全局查询视图
   - 高可用架构

3. **Cortex**：
   - 多租户支持
   - 水平可扩展
   - 长期存储

## 8. 对 ChronoDB 的启示

### 8.1 可借鉴的功能

1. **Recording Rules**：
   - **借鉴价值**：ChronoDB 可以实现类似的预聚合规则机制
   - **实现建议**：
     - 支持自动创建预聚合规则
     - 基于查询频率自动识别高频查询
     - 定期评估预聚合规则并存储结果
   - **预期效果**：减少运行时负载，提高查询性能

2. **服务发现机制**：
   - **借鉴价值**：ChronoDB 可以支持更多的服务发现类型
   - **实现建议**：
     - 支持 Kubernetes 服务发现
     - 支持 Consul 服务发现
     - 支持 DNS 服务发现
   - **预期效果**：简化配置管理，提高运维便利性

3. **Alertmanager**：
   - **借鉴价值**：ChronoDB 可以实现类似的告警管理功能
   - **实现建议**：
     - 实现告警去重、分组、路由
     - 支持静默和抑制
     - 支持多种通知渠道
   - **预期效果**：完善告警管理能力，提高用户体验

4. **联邦架构**：
   - **借鉴价值**：ChronoDB 可以实现更简单的联邦架构
   - **实现建议**：
     - 简化联邦配置
     - 自动数据聚合
     - 支持跨服务监控
   - **预期效果**：解决大规模监控问题，提高可扩展性

### 8.2 可优化的点

1. **查询性能优化**：
   - **问题**：Prometheus 单线程 PromQL 执行，查询性能有限
   - **优化方向**：
     - 实现查询并行化
     - 引入查询优化器
     - 支持向量化执行
   - **预期效果**：查询性能提升 10 倍以上

2. **存储效率优化**：
   - **问题**：Prometheus Gorilla 压缩比有限，无列式存储
   - **优化方向**：
     - 实现列式存储
     - 采用更高效的压缩算法（ZSTD + 预测编码）
     - 支持数据分层
   - **预期效果**：存储效率提升 10 倍以上

3. **降采样机制**：
   - **问题**：Prometheus 不支持内置降采样
   - **优化方向**：
     - 实现多级降采样（L0-L4）
     - 查询时自动选择最优精度
     - 支持降采样数据持久化
   - **预期效果**：长时间段查询性能提升 100 倍以上

4. **分布式架构**：
   - **问题**：Prometheus 无内置分布式支持
   - **优化方向**：
     - 原生支持分布式架构
     - 自动数据分片
     - 支持副本管理
   - **预期效果**：支持水平扩展，解决大规模监控问题

### 8.3 技术方向建议

1. **短期规划（1-3 个月）**：
   - 实现 Recording Rules 功能
   - 优化查询性能（查询并行化、向量化执行）
   - 完善告警管理功能

2. **中期规划（3-6 个月）**：
   - 实现多级降采样机制
   - 优化存储效率（列式存储、高级压缩）
   - 支持服务发现机制

3. **长期规划（6-12 个月）**：
   - 实现分布式架构
   - 简化联邦架构
   - 支持长期存储和全局查询

## 9. 总结

Prometheus 作为云原生监控的事实标准，具有成熟的生态系统、强大的查询语言和完善的告警管理能力。然而，其在单节点限制、查询性能、存储效率和分布式能力方面存在明显不足。

ChronoDB 可以借鉴 Prometheus 的 Recording Rules、服务发现机制、Alertmanager 和联邦架构等优秀功能，同时在查询性能、存储效率、降采样机制和分布式架构方面进行优化，实现 10 倍性能提升和 10 倍存储成本降低的目标。

通过深入分析 Prometheus 的优势和劣势，ChronoDB 可以明确技术发展方向，在保持 Prometheus 兼容性的同时，实现更高的性能和更低的成本。
